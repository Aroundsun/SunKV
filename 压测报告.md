# SunKV 压测报告（基于 redis-benchmark）

## 1. 压测目标

- 使用 Redis 官方压测工具 `redis-benchmark` 对 SunKV 服务端进行基准测试。
- 验证在 RESP 命令路径下的吞吐与稳定性表现。

## 2. 压测环境

- 时间：2026-04-18
- 系统：Linux（本机）
- 压测工具：`redis-benchmark 8.0.2`
- 服务端：`./build/sunkv --port 6379`
- 客户端连通性校验：`./build/sunkvClient 127.0.0.1 6379 PING` 返回 `PONG`

## 3. 压测命令

由于 `redis-benchmark` 默认会尝试 `PING_INLINE`，而 SunKV 当前主路径为 RESP 数组协议，故采用更贴近实际的命令压测并加超时保护：

```bash
timeout 15 redis-benchmark -h 127.0.0.1 -p 6379 -t set -n 50000 -c 50 -q
timeout 15 redis-benchmark -h 127.0.0.1 -p 6379 -t get -n 50000 -c 50 -q
timeout 15 redis-benchmark -h 127.0.0.1 -p 6379 -t set,get -n 50000 -c 50 -P 16 -q
```

## 4. 关键观测

### 4.1 工具兼容性提示

- 多次出现：`WARNING: Could not fetch server CONFIG`
- 说明 SunKV 未实现 Redis 的 `CONFIG` 相关语义（不影响核心读写路径本身，但会影响 `redis-benchmark` 的部分自动探测流程）。

### 4.2 GET 压测（c=50）现象

压测输出（节选）：

- 初始阶段吞吐可达约 `63k ~ 70k rps`
  - 示例：`GET: rps=70516.0`
  - 示例：`GET: rps=65372.0`
- 随后出现长期 `rps=0.0`，测试无法在超时时间内自然结束，需由 `timeout` 强制中断。

### 4.3 SET/PIPELINE 压测现象

- `SET` 在前段有稳定吞吐（约 `10k~11k rps`），随后同样进入 `rps=0.0` 持续阶段。
- `set,get + pipeline(P=16)` 场景也出现相同停滞，未在限定时间内完成。

## 5. 结论

### 5.1 当前可确认结论

- SunKV 在压测初段能处理较高并发请求（至少能观测到 `10k+ ~ 70k rps` 区间）。
- 在 `redis-benchmark` 持续压测中，服务表现出“前段有吞吐、后段停滞（rps=0）”的问题，导致压测无法完整收敛。

### 5.2 当前风险判断

- 该问题更接近“高压持续阶段稳定性/兼容性问题”，而非“完全不可用”。
- 需要进一步定位：连接生命周期、输出缓冲、benchmark 协议细节兼容（尤其是 benchmark 的请求/统计机制）等路径。

## 6. 建议的下一步排查

1. 在压测期间打开服务端关键计数日志（连接数、命令数、异常数、关闭原因）。
2. 单独复现 `set` 与 `get` 场景（去掉混合命令），逐步提升 `-c` 与 `-P` 观察临界点。
3. 对 `redis-benchmark` 用例做协议白名单（仅跑 SunKV 已支持且已验证的命令集）。
4. 增加一个“长时间高并发稳定性”集成测试，复用同一命令模式持续压测并检测是否出现停滞。

---

本报告以本机一次实测为准，属于“现状基线报告”；后续可在修复后对比复测并补充趋势图。
