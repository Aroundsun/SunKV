# SunKV 信号优雅关闭修复记录

## 1. 问题现象

在服务端运行期间，发送 `SIGINT` 或 `SIGTERM` 后，进程有概率不退出，表现为：

- 客户端功能可正常访问；
- 发送终止信号后服务端仍驻留；
- 最终需要 `SIGKILL` 才能强制结束。

---

## 2. 根因分析

原实现中，信号处理函数 `signalHandler(int)` 直接调用了：

- `g_server->setStopping()`
- `g_server->stopMainLoop()`

这类调用涉及 C++ 对象方法与复杂运行时状态，不属于异步信号安全操作。  
在信号上下文直接执行，行为不可靠，可能导致停机流程无法稳定触发。

---

## 3. 修复方案

采用“**信号最小处理 + 管道转发 + 普通线程执行业务逻辑**”方案。

### 3.1 信号处理函数最小化

文件：`server/Server.cpp`

- `signalHandler(int)` 仅执行异步信号安全动作：向管道写入 1 字节信号通知。
- 不再直接调用任何 `Server` 成员方法。

### 3.2 引入信号转发线程

文件：`server/Server.h`、`server/Server.cpp`

新增成员：

- `std::thread signal_thread_`
- `std::atomic<bool> signal_thread_running_{false}`

逻辑：

1. `start()` 中启动信号线程；
2. 信号线程阻塞读取管道；
3. 收到字节后在普通线程上下文调用：
   - `setStopping()`
   - `stopMainLoop()`

### 3.3 停机时线程回收

文件：`server/Server.cpp`

- `stop()` 中先将 `signal_thread_running_` 置为 `false`；
- 再向管道写入唤醒字节，打断阻塞 `read`；
- `join()` 等待信号线程退出，避免线程泄露。

---

## 4. 涉及文件

- `server/Server.cpp`

## 5. 验证过程

验证步骤：

1. 编译服务端：`cmake --build build --target sunkv`
2. 启动服务端：`./build/sunkv`
3. 获取实际服务端进程 PID
4. 发送 `SIGINT`
5. 观察进程退出与日志链路

验证结果：

- 服务端收到 `SIGINT` 后稳定退出；
- 日志显示完整优雅关闭路径：
  - 主循环退出
  - 执行 `Server::stop()`
  - 停止网络与线程池
  - 快照与 WAL 同步
  - 存储清理完成
- 退出码为 `0`。

---

## 6. 结论

本次修复将不安全的信号上下文调用迁移到安全线程上下文执行，解决了 `SIGINT/SIGTERM` 下服务端无法稳定优雅退出的问题。  
方案符合常见服务器信号处理最佳实践，后续可复用于更多异步事件通知场景。
