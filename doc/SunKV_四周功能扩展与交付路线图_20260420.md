# SunKV 四周功能扩展与交付路线图（校招 C++ 向）

> 版本：2026-04-20  
> 目标读者：项目作者（执行）+ 面试官（评估）  
> 适用范围：当前 SunKV 主干（已完成 pipeline 逻辑修复与压测对比）

---

## 1. 项目目标与定位

### 1.1 目标
在 4 周内，把 SunKV 从“基础 KV + 持久化 + 网络框架”升级为“具备事务、发布订阅、可观测性与稳定交付能力”的可展示项目。

### 1.2 校招展示核心卖点
- 协议与状态机：RESP 命令解析、连接态管理（事务态/订阅态）
- 并发与网络：多连接模型、背压控制、事件循环
- 存储与一致性：命令执行顺序、WAL/快照恢复语义
- 工程质量：压测复现、CI 门禁、文档闭环

### 1.3 成功标准（最终验收）
- 功能：`MULTI/EXEC/DISCARD` + `SUBSCRIBE/PUBLISH` 可用
- 质量：关键路径有集成测试，异常路径可复现
- 性能：`GET/SET` 在 `P=16` 下继续保持正收益，新增功能不明显拖垮基础路径
- 文档：可按文档命令一键复现实验和验证

## 3. 第 1 周：事务子集（详细到日）

### Day1：设计与边界
- 明确支持命令：`MULTI`、`EXEC`、`DISCARD`
- 明确不支持项（先显式报错）：`WATCH`、嵌套事务
- 在文档中写清事务语义：
  - `MULTI` 后普通写命令入队，不立即执行
  - `EXEC` 批量执行并返回数组结果
  - `DISCARD` 丢弃队列并退出事务态

验收：
- 事务状态流转图完成（Idle -> InTxn -> Exec/Discard -> Idle）

### Day2：连接上下文改造
- 在连接上下文增加：
  - `in_transaction` 标志
  - `queued_commands` 队列（存储已解析命令）
- 改命令分发入口：事务态下除控制命令外一律入队

验收：
- 编译通过
- 非事务路径行为不变

### Day3：`EXEC` 执行器
- 实现队列顺序执行器
- 每条命令结果聚合为 RESP 数组返回
- 单条命令错误不导致服务崩溃，按结果项返回错误

验收：
- `MULTI -> SET -> GET -> EXEC` 正确返回数组

### Day4：异常与一致性处理
- `EXEC` 空队列返回空数组
- `DISCARD` 清空队列并退出事务态
- 重连/断连后事务状态自动释放（连接级别）

验收：
- 异常路径覆盖：重复 `MULTI`、`DISCARD` 非事务态、`EXEC` 非事务态

### Day5：测试与文档
- 新增事务集成测试
- 补充文档：支持矩阵、限制项、示例命令
- 跑回归：基础 `SET/GET` + pipeline 快速回归

验收：
- CI 本地通过（至少构建 + 核心测试）

#### 事务支持矩阵（当前实现）
| 命令 | 状态 | 行为 |
|---|---|---|
| `MULTI` | 已支持 | 进入事务态，返回 `+OK`；嵌套调用返回 `-ERR MULTI calls can not be nested` |
| `EXEC` | 已支持 | 非事务态返回 `-ERR EXEC without MULTI`；事务态按入队顺序执行并返回 RESP 数组 |
| `DISCARD` | 已支持 | 非事务态返回 `-ERR DISCARD without MULTI`；事务态清空队列并返回 `+OK` |
| `WATCH` / `UNWATCH` | 不支持（显式报错） | 返回 `-ERR WATCH/UNWATCH is not supported` |

#### 事务限制项（当前版本）
- 仅支持最小闭环事务：`MULTI/EXEC/DISCARD`。
- 不支持 `WATCH/UNWATCH`，并显式返回错误。
- 不支持嵌套事务。
- `EXEC` 中单条命令失败只影响该结果项，不回滚已执行命令（与 Redis 一致）。

#### 示例命令（可直接复现）
```bash
# 1) 正常事务
redis-cli -p 6379 MULTI
redis-cli -p 6379 SET a 1
redis-cli -p 6379 GET a
redis-cli -p 6379 EXEC

# 2) 异常路径
redis-cli -p 6379 EXEC
redis-cli -p 6379 DISCARD
redis-cli -p 6379 WATCH a
```

## 4. 第 2 周：发布订阅（详细到日）

### Day1：模型设计
- 数据结构：
  - `channel -> subscribers(set<conn>)`
  - `conn -> channels(set<string>)`（便于断连清理）
- 协议行为定义：
  - `SUBSCRIBE` 后进入订阅态（最小实现可限制只允许订阅相关命令）

### Day2：`SUBSCRIBE/UNSUBSCRIBE`
- 支持单/多 channel 订阅与取消
- 返回订阅确认消息（兼容 Redis 基本格式）

### Day3：`PUBLISH`
- `PUBLISH channel msg` 广播给频道订阅者
- 返回投递数量

### Day4：可靠性处理
- 连接关闭时清理订阅关系
- 发送失败/背压时不影响全局状态一致性

### Day5：测试与文档
- 多连接订阅广播测试
- 断连清理测试
- 文档补充：订阅态行为与边界说明

## 8. 风险清单与应对

### 风险 1：功能堆叠导致主干回归
- 应对：每周固定“基础路径回归”作为门禁（`SET/GET P=1/P=16`）

### 风险 2：协议兼容细节过深，时间失控
- 应对：先做“最小子集 + 明确不支持项”，避免无限扩张

### 风险 3：文档滞后，成果不可展示
- 应对：每个功能 PR 同步更新文档，不允许“后补”

### 风险 4：改动过大难以定位问题
- 应对：按主题小提交，保持可回滚
