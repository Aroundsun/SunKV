# 项目双轨问题分析报告（SunKV）

本文档用于系统性记录 SunKV 当前存在的“同一类能力存在两套实现/两条路径/两套语义”的**双轨问题**，并给出证据、影响评估、来源判断（历史遗留/近期遗漏）与收敛建议。

更新时间：2026-04-13（已按近期改动复核更新）

---

## 背景与判定标准

### 什么是“双轨问题”

当同一项工程能力（例如：日志、配置、恢复、事件模型、压测口径）在代码/脚本/文档中出现两套并行实现或两条不一致路径，并且：

- **用户/运维/开发者难以判断实际生效的是哪一条**；或
- **不同路径得到的行为/数据不一致**；或
- **排障与压测结果被割裂/扭曲**；

就视为需要收敛的双轨问题。

### 本报告目标

- 找出对**生产行为**、**压测可比性**、**排障效率**影响最大的双轨点；
- 为后续整改提供“可落地的收敛方向”（短期/长期）；
- 避免在修 bug/压测阶段持续叠加补丁导致系统更难理解。

## 双轨问题清单（高价值项）

> 说明：每条包含 **证据**（路径/符号/描述）、**影响**、**判断**（历史遗留 vs 近期遗漏）、**建议**（短期/长期）。

### 1) 日志输出双轨：`spdlog/LOG_*` vs `std::cout/std::cerr/perror`

- **状态**：**部分已收敛（运行期代码已统一，测试/客户端仍允许使用 stdio）**
- **证据**
  - `network/logger.h`：`LOG_DEBUG/LOG_INFO/LOG_WARN/LOG_ERROR` 宏基于 `Logger::instance().getLogger()`
  - 运行期目录（`server/ network/ storage/ common/ protocol/ command/`）已通过 `ctest` 用例 **`NoStdioRuntimeTest`** 约束：禁止 `std::cout/std::cerr/perror/printf`
  - 仍存在 `std::cout/std::cerr` 的位置主要在测试与工具侧：
    - `test/*.cpp`（测试输出/断言信息）
    - `client/simple_client.cpp`（交互客户端）
- **影响**
  - **排障割裂**：同一次运行信息分散在 spdlog 文件、stdout、stderr；容器/脚本采集只抓一种会漏关键线索。
  - **性能扭曲**：高并发输出到 stdout/stderr 会显著影响延迟/吞吐；即使 spdlog 控制了级别，`cout/cerr` 仍可能刷屏。
- **判断**
  - **历史遗留 + 补丁混用**：spdlog 已成体系，但残留大量直接输出。
- **建议**
  - **短期（已执行）**：运行期代码禁止 `std::cout/std::cerr/perror/printf`，统一改 `LOG_*`。
  - **长期（已部分执行）**：通过 `scripts/no_stdio_runtime_check.sh` + `ctest` 用例做静态检查；可进一步升级为 CI 强制规则（PR 失败）。

### 2) 日志文件策略双轨：logger 默认路径 vs `main` 的“每次运行独立文件”

- **状态**：**已收敛（策略下沉到 Logger/配置层，main 不再拼路径）**
- **证据**
  - `network/logger.cpp`：由 `Logger::setFileStrategy()` 统一控制实际落盘路径（`fixed/per_run/daily`），并在进程内固定 resolved 路径
  - `common/Config.h`：`log_file` 默认 `./data/logs/sunkv.log`
  - `common/Config.h`：新增 `log_strategy`（`fixed|per_run|daily`）
  - `server/main.cpp`：仅负责接线（`setFileStrategy(config.log_strategy)` + `setFile(config.log_file)`），不再手工拼接时间戳路径
- **影响**
  - **运维预期不一致**：以为固定写 `sunkv.log`，实际每次运行换文件名；采集规则/路径匹配可能失效。
  - **策略叠加**：按次运行分文件 + rotating_file_sink（文件内滚动）叠加，理解成本上升。
- **判断**
  - **近期增强未收敛**：main 层增强了“按次运行文件”，但 logger/配置层未形成统一策略开关。
- **建议**
  - **短期（已执行）**：日志文件策略统一由 `log_strategy` 控制，默认 `fixed`。
  - **长期（已执行）**：命名/归档策略下沉到 `Logger`/配置层；后续可补充策略的 schema 校验与更完善的文档示例。

### 3) 压测/脚本层面的日志采集双轨：`server_stdout.log` vs spdlog 文件

- **状态**：**仍存在（脚本重定向与 spdlog 文件并存）**
- **证据**
  - `scripts/perf_profile.sh`：启动服务时重定向 stdout/stderr 到 `data/logs/*_server_stdout.log`
  - `scripts/stability_suite.sh`：同样重定向到 `${LONG_LOG}`
  - 同时程序内部仍写 spdlog 文件（`log_file`），且历史上存在 `cout/cerr` 输出路径。
- **影响**
  - **报告不完整**：只看 `server_stdout.log` 可能漏掉 spdlog 文件里的关键错误/统计；只收 spdlog 也可能漏掉 stderr。
- **判断**
  - **历史遗留**：脚本为了收敛输出；程序又“自产生日志文件”形成双写/双源。
- **建议**
  - **短期**：压测脚本明确“唯一真相日志来源”（推荐统一收集 spdlog 文件，并让启动阶段也走 spdlog）。
  - **长期**：提供 `--log-to-stdout/--log-to-stderr` 模式，使脚本通过重定向采集而不是“双写”。

### 4) 配置文件语义双轨：保存像 INI（有 `[section]`）但解析是扁平键

- **证据**
  - 历史上 `common/Config.cpp::saveToFile()` 写过 `[network] / [storage] / ...` 段落（外观像 INI）
  - 解析始终是扁平 `key=value`：`Config::parseLine()` 不处理 section
  - 现在 `saveToFile()` / `generateSampleConfig()` 已收敛为 **扁平键 + 分组注释**，并在 `loadFromFile()` 中显式忽略 `[section]`
- **影响**
  - **误配置风险高**：配置看起来像 INI，但 section 实际无语义；用户可能按 INI 习惯改错键名而不生效。
- **判断**
  - **典型历史遗留双轨**：外观像 INI、语义却不是 INI。
- **建议**
  - **短期**：明确选择一种格式：要么把保存格式改为“无 section 的扁平键”，要么让解析器真正支持 section 并同步 `applyConfig`。
  - **长期**：引入 schema（键名/类型/默认值/来源优先级），自动生成 usage + sample config，消灭“看似支持实际不支持”的配置。（已落地：见 `common/Config.cpp` 的 schema 注册表与自动生成）

---

### 5) 配置来源双轨：文件走 `config_map_`，CLI 直接改成员变量（且 `--config` 可能在 argv 中途触发）

- **证据**
  - 旧实现中 `Config::loadFromArgs()` 支持 `--config <file>` 在 argv 中途触发再次加载，产生顺序依赖
  - 现已收敛为固定管线：`default → (--config 指定文件，可选且只加载一次) → CLI`（`Config::loadFromArgs()` 前置扫描 `--config`）
- **影响**
  - **优先级/顺序依赖**：`--config` 出现在 argv 中间会出现“先 CLI 覆盖一部分 → 读文件覆盖 → 再 CLI 覆盖”的顺序依赖，定位困难。
  - **生效确认困难**：缺乏“最终配置摘要 + 来源（default/file/cli）”输出。
- **判断**
  - **近期遗漏/演进中**：已有 `log_level_from_cli` 这种来源标记，但还没系统化覆盖所有字段。
- **建议**
  - **短期**：固定管线：default → config file → CLI（只允许一次 config 加载且顺序固定）。
  - **长期**：在启动日志打印一次性“最终配置摘要 + 每项来源”，快速确认配置是否生效。（已落地：`Config::dumpEffectiveConfigWithSource()` + `main` 启动日志输出）

### 6) 事件模型错配风险（高优先级）：`epoll` 驱动但 `Channel` 用 `poll` 常量解析 `revents_`

- **证据**
  - `network/Poller.*`：使用 `epoll_create1/epoll_wait/epoll_ctl`
  - `network/Poller.cpp`：`channel->setRevents(events_[i].events)` 把 epoll events 位图塞给 Channel
  - `network/Channel.cpp`：包含 `<poll.h>`，在 `handleEventWithGuard()` 使用 `POLLIN/POLLOUT/POLLERR/POLLHUP/...` 判断 `revents_`
- **影响**
  - **生产级严重风险**：epoll 与 poll 位图并非承诺一致；边界位（例如 `EPOLLRDHUP`）可能被误判，导致连接泄漏、忙轮询、丢事件或错误关闭。
- **判断**
  - **强历史遗留特征**：像从 poll 迁到 epoll，但 Channel 的事件位解释没有完全迁移。
- **建议**
  - **短期**：统一事件位体系：
    - 方案 A：Channel 全部改用 `EPOLL*` 常量解释；
    - 方案 B（推荐）：Poller 将 epoll events 映射为 Channel 内部抽象位（read/write/error/close），Channel 只看抽象位。
  - **长期**：为 `HUP/ERR/RDHUP` 等边界加入单测/回归用例，保证未来不再“隐性错配”。

### 7) 停止/析构路径双轨：`main` 显式 stop + `Server::~Server` stop + 信号线程触发 quit

- **证据**
  - `server/main.cpp`：主循环退出后执行 `server->stop()`
  - `server/Server.cpp`：`Server::~Server()` 内调用 `stop()`
  - `server/Server.cpp`：信号线程在 SIGINT/SIGTERM 下设置 stopping/quit
  - `Server::stop()` 里存在 `static std::atomic<bool> shutdown_executed` 防重入逻辑
- **影响**
  - **生命周期复杂**：多条路径竞争 stop/quit，靠“防重入”会掩盖“第一次 stop 是否完整执行”的事实。
  - **线上/测试差异**：不同触发方式（kill、析构、main 退出）走不同组合，导致偶发泄漏/卡住/半关闭。
- **判断**
  - **后期修火补丁叠加**：出现“全局防重复关闭”通常表示生命周期设计未收敛。
- **建议**
  - **短期**：定义唯一 stop 入口；析构只做兜底；main 不重复 stop。
  - **长期**：关闭流程改为“阶段机幂等”，每阶段最多一次且记录阶段进度，避免二次 stop 直接 return 掩盖问题。

### 8) 恢复（replay）实现双轨：`Server` 内部恢复 vs `storage/Recovery.*` 独立恢复器

- **证据**
  - `server/Server.cpp`：初始化持久化时“快照→（若快照加载成功则跳过 WAL）/否则 WAL replay”
  - `storage/Recovery.h/.cpp`：`DataRecovery::recover_data(...)` 走“快照→WAL→一致性检查”的通用流程；默认目录与 `Config` 的 `./data/...` 可能不一致
- **影响**
  - **恢复口径不一致**：一个实现“有快照就完全跳过 WAL”，另一个实现“快照后仍 replay WAL”，可能导致崩溃恢复后的数据差异。
  - **目录默认值分裂**：恢复到空目录仍可能“看似成功”。
- **判断**
  - **强历史遗留双轨**：两套恢复器并存但主路径未统一。
- **建议**
  - **短期**：明确唯一恢复策略（建议：快照恢复后仍 replay WAL，并按快照序列号/时间戳过滤）。
  - **长期**：收敛到一个模块（Server 调用 DataRecovery 或删除/下线 DataRecovery），并统一目录来源于 `Config`。

---

### 9) 多类型持久化“名义支持”双轨：接口存在，但当前加载路径可能退化为字符串

- **证据**
  - `server/Server.*`：存在 `create_multi_type_snapshot/load_multi_type_snapshot` 等接口
  - 当前加载路径仍可能走 `snapshot_manager_->load_snapshot(std::map<string,string>)` 并 `DataValue(string)`，导致类型退化
  - `storage/Snapshot.*`：多类型可能采用“拼字符串”编码，缺少对称反解析
- **影响**
  - **语义风险**：恢复后类型退化/不可逆编码，影响生产语义与一致性。
- **判断**
  - **近期扩展未补齐对称性**：接口先铺开，读回/编码层未收敛。
- **建议**
  - **短期**：快照落盘写入 type + 长度等自描述信息，实现对称反序列化。
  - **长期**：统一 `DataValue` 的稳定二进制编码（快照/WAL 共享编码层），避免再出现“写入可用、恢复退化”的双轨。

---

### 10) 压测/稳定性脚本口径双轨：端口/参数/模式不一致导致结果不可比

- **证据**
  - `scripts/perf_profile.sh`：偏 perf/strace profiling；默认端口/并发/请求数不同
  - `scripts/stability_suite.sh`：偏稳定性与恢复验证；默认端口/连接上限/并发/恢复模式不同
- **影响**
  - **结果不可比**：不同脚本之间 QPS/延迟差异巨大，容易误判优化收益或引入回退。
- **判断**
  - **阶段性脚本各自演进**的历史遗留。
- **建议**
  - **短期**：抽公共 `bench_env.sh`（端口、并发、连接、pipeline、数据规模、build type、log level）供脚本 source，保证口径可比。
  - **长期**：建立基准测试规范并在输出中打印“实际启动参数/配置来源/构建类型/日志级别”。

## 优先级建议（按收益与风险排序）

- **P0（最高优先级）**
  - **事件模型错配（第 6 条）**：可能直接导致线上连接/事件异常，且问题隐蔽，建议优先修复并加回归测试。

- **P1（高优先级）**
  - 日志输出/采集双轨（第 1/2/3 条）：直接影响排障与压测可信度。
  - 配置语义/来源双轨（第 4/5 条）：直接影响“配置是否生效”的可预期性。
  - 恢复逻辑双轨（第 8/9 条）：直接影响数据一致性与崩溃恢复正确性。
  - 脚本口径双轨（第 10 条）：直接影响性能结论可比性与复现性。

## 建议的整改路线（可选）

### 里程碑 A：先消除生产级风险
- 修复事件位体系（第 6 条）：统一 EPOLL 事件解释或映射到抽象位，并加测试覆盖。

### 里程碑 B：统一可观测性与可配置性
- 日志统一（第 1/2/3 条）：输出源统一、文件策略统一、脚本采集统一。
- 配置统一（第 4/5 条）：配置格式与解析语义一致；固定加载顺序；输出配置摘要与来源。

### 里程碑 C：恢复与多类型对称
- 收敛恢复器与落盘编码（第 8/9 条）。

### 里程碑 D：基准口径统一
- 统一压测脚本参数源（第 10 条）。
