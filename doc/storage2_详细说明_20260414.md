# storage2（SunKV 存储模块 v2）详细说明（落地版）

> 本文是对 `storage2/` 现有实现的“代码对齐说明”，偏工程落地与使用方式，不再重复旧 `storage/` 的历史问题。
>
> 目标：让你可以在不读全量代码的情况下，清楚理解 v2 的边界、调用契约、持久化链路与组装方式，并知道目前的限制与下一步怎么接入 `Server`。

---

## 1. 总体设计（你关心的“解耦点”）

`storage2` 采用 **单向依赖** 分层，避免旧架构的耦合/双轨：

- **Server / 协议层**：只依赖 `storage2/api/IStorageAPI.h`
- **命令语义层（API）**：定义 Redis-like 语义与统一返回值，不认识 RESP
- **执行引擎层（engine）**：实现命令语义、TTL 惰性删除、类型判断/转换
- **后端层（backend）**：只负责 `Record` CRUD，不知道命令语义
- **持久化层（persistence）**：只认识 `Mutation/Record` 的编码与回放，不散落在命令分支里
- **装饰器层（decorators）**：cache/metrics/自动持久化等横切能力，按 wiring 组合，不侵入核心

关键约束：

- **存储层不认识 RESP**：不返回协议串，只返回结构化 `StorageResult<T>`
- **持久化层不认识命令**：只消费 `MutationBatch`（Put/Del/ClearAll）
- **缓存不改变语义**：TTL/类型错误必须与底层一致（v0.1 采用“安全缓存”策略）

## 2. 目录结构与职责

### 2.1 `storage2/api/`

- `StatusCode.h`
  - 统一状态码：`Ok/NotFound/WrongType/InvalidArg/InternalError/NotImplemented`
- `StorageResult.h`
  - `StorageResult<T> { status, value, mutations }`
  - `mutations` 是本次操作产生的写事件（给持久化层用）
- `IStorageAPI.h`
  - Redis-like 命令语义接口（strings/lists/sets/hashes/ttl/keys/dbsize）

### 2.2 `storage2/engine/`

- `Mutation.h`
  - **Record**：唯一真相 `DataValue + expire_at_us + version`
  - **Mutation**：写入事件（PutRecord/DelKey/ClearAll）
- `Time.h/.cpp`
  - `nowEpochUs()`：统一时间来源（TTL 统一用 epoch us）
- `StorageEngine.h/.cpp`
  - `IStorageAPI` 的核心实现（string/list/set/hash/ttl/keys/dbsize）
  - TTL 策略：**惰性删除**（访问时检查过期，过期则删除并视为 NotFound）
  - 类型转换策略：**Redis-like**（当 key 为 STRING 且执行容器写命令时，转为目标容器且不注入旧字符串）
  - 恢复入口（引擎级）：
    - `loadSnapshot(records)`
    - `applyMutation(m)`（支持 Put/Del/ClearAll）

### 2.3 `storage2/backend/`

- `IBackend.h`
  - 最小 CRUD：`getRecord/putRecord/delKey/clearAll/size/keys`
- `InMemoryBackend.h/.cpp`
  - 单机内存 `unordered_map` + mutex
- `ShardedBackend.h/.cpp`
  - 真分片：N 个独立 backend 实例（当前用 FNV-1a 稳定哈希分片）

### 2.4 `storage2/persistence/`

- `RecordCodec.h/.cpp`
  - 对 `Record` 做稳定二进制编码
  - 复用现有 `storage/DataValueCodec`（稳定 DataValue 编解码）
- `WalCodec.h/.cpp`
  - WAL 记录的编码/解码（读写同源）
  - 支持三态解码：`Ok / IncompleteTail / Corrupt`
    - `IncompleteTail`：容忍崩溃导致尾部半条记录
- `WalWriter.h/.cpp`
  - 负责 `write()` / `fsync()`，支持 `appendBytes()`（一次写入已编码 buffer）
- `WalReader.h/.cpp`
  - 读取 WAL 文件并用 `WalCodec` 解码（容忍尾部半条）
- `SnapshotWriter.h/.cpp`
  - 全量快照写出：遍历 backend 的 `keys()`，写入 key+RecordCodec bytes
- `SnapshotReader.h/.cpp`
  - 读回全量快照
  - 同时提供 `readFromFile()`（只产出 records，不直接写 backend）
- `PersistenceOrchestrator.h/.cpp`
  - 持久化编排器（v0.1 可用能力）：
    - 接收 `MutationBatch`
    - 异步队列（`condition_variable`，避免空转）
    - WAL group commit（可调阈值：条数/字节/linger）
    - 刷盘策略骨架：Never / Always / Periodic
    - `flush()`：**drain 队列并落盘**（测试/退出路径用）
    - `recoverInto(StorageEngine&)`：Snapshot -> WAL 回放到引擎（由引擎落到 backend）

### 2.5 `storage2/decorators/`

- `CachedStorageAPI.h/.cpp`
  - **安全缓存**：只缓存 `GET` 的“无 TTL”（`PTTL==-1`）结果；写操作 invalidate(key)
  - 目的：先保证语义正确，不抢跑复杂 LRU/LFU/ARC
- `MetricsStorageAPI.h/.cpp`
  - 统计调用次数与 `StatusCode` 分布（轻量 atomic 计数）
- `OrchestratedStorageAPI.h/.cpp`
  - 自动把 `StorageResult.mutations` 提交给 `PersistenceOrchestrator`
  - 并把返回给上层的 `mutations` 清空（避免重复提交）

### 2.6 `storage2/Factory.h/.cpp`

提供“一键 wiring”能力：

- 选择 backend：`shard_count==1` 用 InMemory；`>1` 用 Sharded(InMemory...)
- 创建 engine
- 选择性包上 cache/metrics
- 选择性创建 orchestrator（WAL/恢复）
- 若存在 orchestrator：最外层包 `OrchestratedStorageAPI`（上层无需手动 submit）

## 3. 核心数据模型：Record 与 Mutation

### 3.1 Record

`Record` 是 backend 层存储的最小单位：

- `value: DataValue`（STRING/LIST/SET/HASH）
- `expire_at_us: int64_t`
  - `-1` 表示无 TTL
  - 统一用 epoch 微秒，避免旧实现 steady_clock 的持久化歧义
- `version: uint64_t`
  - v0.1 先作为占位（未来可做并发/一致性增强）

### 3.2 Mutation

写操作必须产出写事件：

- `PutRecord`：key -> record（包含 value+expire_at）
- `DelKey`：删除 key
- `ClearAll`：清空（当前还未暴露成 API 命令，但恢复/工具可用）

持久化层只消费这些 mutation，而不是分散在命令分支里直接写 WAL。

---

## 4. 统一返回：StorageResult<T>

所有 API 都返回 `StorageResult<T>`：

- `status`：统一状态码
- `value`：具体返回值（如 `int64_t`、`optional<string>` 等）
- `mutations`：写事件批（给持久化用）

注意：

- **如果启用 `OrchestratedStorageAPI`**：对上层返回的 `StorageResult`，`mutations` 会被清空。
  - 原因：mutation 已被自动提交给 orchestrator；上层再提交会造成重复落盘。

---

## 5. 命令语义（v0.1 已实现范围）

接口见 `storage2/api/IStorageAPI.h`，实现见 `storage2/engine/StorageEngine.cpp`。

### 5.1 String

- `SET/GET/DEL/EXISTS`
- WrongType：对非 STRING 的 `GET` 返回 `WrongType`

### 5.2 TTL

- `EXPIRE/PERSIST/TTL/PTTL`
- 过期策略：惰性删除（读/写都会触发检查并删除）

### 5.3 List / Set / Hash

- list：`LPUSH/RPUSH/LPOP/RPOP/LLEN`
- set：`SADD/SREM/SCARD/SISMEMBER/SMEMBERS`
- hash：`HSET/HGET/HDEL/HLEN/HEXISTS/HGETALL`

### 5.4 管理

- `DBSIZE/KEYS`
  - v0.1 `KEYS` 直接返回 backend `keys()`（不做 scan，不过滤过期；过期在访问时惰性删除）

## 6. 类型转换契约（避免你之前踩过的坑）

当 key 现有类型为 STRING，但执行容器写命令（如 `LPUSH/SADD/HSET`）时：

- v0.1 采用 Redis-like 行为：转换成目标容器类型
- **不会把旧 `string_value` 注入到容器里**

这条契约在 v0.1 的 `StorageEngine::getOrInitContainer()` 中强制执行。

---

## 7. 持久化闭环（WAL + Snapshot）

### 7.1 RecordCodec

`RecordCodec` 对 Record 做稳定二进制编解码：

- 内部调用 `storage/DataValueCodec` 编码 value（并携带 expire_at_us）

### 7.2 WalCodec / WalWriter / WalReader

WAL 以 `Mutation` 为单位追加：

- `WalCodec` 是唯一的读写协议定义（避免漂移）
- `WalReader` 解码时：
  - **容忍尾部半条（IncompleteTail）**：停止回放并返回成功
  - 对 magic/version/record 解码失败等 **真实损坏（Corrupt）**：返回失败

### 7.3 SnapshotWriter / SnapshotReader

- v0.1 使用全量快照（写出所有 keys 的 record）
- 读取后可通过 `StorageEngine::loadSnapshot()` 应用

### 7.4 PersistenceOrchestrator

- **写入路径**：接收 `MutationBatch` → WAL group commit → 按策略 fsync
- **恢复路径**：Snapshot -> WAL → 回放到 `StorageEngine`
- **flush()**：用于测试/退出路径，语义是“尽力 drain 队列并落盘”

---

## 8. 装饰器（decorators）如何组合

推荐组合顺序（从内到外）：

1. `StorageEngine`
2. `CachedStorageAPI`（可选）
3. `MetricsStorageAPI`（可选）
4. `OrchestratedStorageAPI`（可选，但建议只要启用持久化就启用它）

理由：

- Cache/Metrics 属于横切能力，放外层更自然
- Orchestrated 应该最外层，保证拿到最终 `StorageResult` 的 mutations 统一提交

---

## 9. Factory：最推荐的使用方式

`storage2/Factory` 提供 `createStorage2()`，返回：

- `api`：给 Server 使用的 `IStorageAPI`
- `engine`：用于恢复的 `StorageEngine*`（所有权在 api 链内部）
- `orchestrator`：可选（WAL/恢复/flush）

用法要点：

1) 启动时：
- `auto c = createStorage2(opt);`
- `Server` 只持有 `c.api.get()`（或把所有权移动进 Server）

2) 恢复时：
- `c.orchestrator->recoverInto(*c.engine);`

3) 退出前：
- `c.orchestrator->flush();`

## 11. 当前限制（v0.1 明确承认的不足）

- **KEYS**：目前是全量返回（没有 SCAN/渐进遍历），且不主动过滤过期
- **Cache**：目前是安全策略（只缓存无 TTL 的 GET），未实现 LRU/LFU/ARC
- **持久化一致性**：
  - WAL 没有文件级 header、record length 前缀、CRC（后续可加强）
  - flush() 是“尽力 drain”，不是严格并发线性化保障
- **尚未接入 Server 主链路**：目前 `storage2` 是独立模块与测试闭环

## 12. 下一步（建议优先级）

### P0：接入 Server（一次性替换）

- Server 只依赖 `IStorageAPI`
- 通过 `Factory` 构建 `api`/`orchestrator`
- 在 Server 停止路径调用 `flush()`

### P1：增强 WAL 格式稳健性

- 引入 record length + CRC32
- 支持“单条损坏跳过/隔离”与更强校验

### P1：Scan/Keys 策略

- 用迭代器/游标避免 O(N) 阻塞

### P2：Cache 策略升级

- 把现有安全缓存替换为可插拔 LRU/LFU/ARC，并严格保持 TTL 语义
