# 日志与配置系统问题清单（待办）

本文档用于**记录现状问题**与**后续改进方向**，不在此处直接修改实现（避免在压测/修 bug 阶段引入新变量）。

更新时间：2026-04-13

## 日志系统（`network/logger.*` + `LOG_*` 宏 + `server/*`）

### 已确认的问题

- **`enable_console_log` 配置未生效（死配置）**
  - **现象**：`Config` 里有 `enable_console_log`，配置文件也会保存/读取，但 `Logger` 并未依据该开关添加/移除控制台 sink。
  - **影响**：运维/使用者会误以为能控制是否输出到控制台；实际行为与配置不一致。
  - **建议**：
    - 要么实现「文件 + 控制台」双 sink 并受 `enable_console_log` 控制；
    - 要么删除该配置项（或标记为未实现）。

- **持久化相关模块仍大量使用 `std::cerr` 输出 `DEBUG:`（不进入 spdlog 文件）**
  - **涉及**：`storage/WAL.cpp`、`storage/Snapshot.cpp` 等。
  - **影响**：
    - 日志输出分裂为「spdlog 文件」与「stderr 重定向」，排障需要同时看两处；
    - 对生产/压测环境，stderr 可能被 systemd/journald 或容器收集，带来额外 I/O。
  - **建议**：
    - 逐步迁移为 `LOG_DEBUG/INFO/...`（或提供统一的 debug 开关），先从“启动/恢复/关闭”路径开始，避免热点路径一次性引入大量日志；
    - 对迁移后的日志统一放到可配置级别下（例如默认 INFO，排障时用 `--log-level DEBUG` 打开）。

- **Debug 级别下高频日志过多，可能扭曲性能**
  - **现象**：`Poller::poll()` 等热点路径有大量 `LOG_DEBUG`。
  - **影响**：
    - DEBUG 压测得到的 QPS/延迟不具代表性；
    - 日志文件体积膨胀、I/O 变成瓶颈。
  - **建议**：
    - 生产默认 `INFO` 或更高；
    - Debug 下对热点日志采样（例如每 N 次打印一次），或拆分子 logger（network/业务分开）。

- **日志级别字符串大小写敏感**
  - **现象**：`Logger::setLevelFromName()` 只识别 `DEBUG/INFO/WARN/ERROR` 大写。
  - **影响**：传入 `debug` 会落回默认（当前实现等价于 INFO），易踩坑。
  - **建议**：在解析前做 `toupper()`，或支持更多别名（如 `warning`/`err`）。

### 已做但仍需注意的点（行为说明）

- **Debug 构建默认使用 `DEBUG` 级别**
  - **当前行为**：未显式传 `--log-level` 时，Debug 构建会把 `log_level` 提升为 `DEBUG`，便于看到运行期细节日志。
  - **注意**：这会覆盖配置文件中的 `log_level`（当且仅当未传 `--log-level`）。
  - **建议**：在 README/样例配置中明确说明「Debug 默认 DEBUG，压测请用 Release+INFO」。

- **Release 构建启用 `-DNDEBUG`**
  - **影响**：所有 `assert` 被编译掉（包括 `Channel::~Channel()` 的不变式），线上问题可能不再“立即崩溃”，而是表现为更隐蔽的资源/状态异常。
  - **建议**：对关键不变式在生产路径用显式检查 + 记录错误（而不是只依赖 assert）。

## 配置系统（`common/Config.*` + `sunkv.conf` + CLI）

### 已确认的问题

- **仓库内缺少默认 `sunkv.conf` 样例**
  - **现象**：程序会尝试加载 `./sunkv.conf`，但仓库当前未提供示例文件。
  - **影响**：新用户不知道有哪些配置键；只能靠源码或 `saveToFile()` 输出摸索。
  - **建议**：提供一个 `sunkv.conf.example`（或默认 `sunkv.conf`）并在 README 引导。

- **配置文件“伪 INI”格式：`[section]` 仅作为注释**
  - **现象**：`saveToFile()` 会写 `[network]` 等段落，但解析器 `parseLine()` 只识别 `key=value`，`[xxx]` 行会被忽略。
  - **影响**：与用户预期的 INI 行为不同，但目前不会造成错误（只是“无 section 语义”）。
  - **建议**：在文档中声明这是“无 section 语义的 key=value 文件”；或真正实现 section（工作量更大）。

- **CLI 覆盖不完整**
  - **现状**：已有 `--port/--host/--thread-pool-size/--max-connections/--data-dir/--config/--log-level/...`
  - **缺口**：例如 `--log-file`、`--enable-console-log`、WAL/快照路径与开关等只能通过配置文件设置。
  - **影响**：容器化/脚本化部署不够友好；调参需要改文件而不是命令行。
  - **建议**：补齐常用 CLI 参数，或提供环境变量覆盖策略（选一种统一即可）。

- **`loadFromFile()` 失败时的处理不够强**
  - **现象**：打开/解析失败会返回 `false`，但 `main` 中对默认配置加载没有强制终止（仅 `exists` 时调用，不检查返回值）。
  - **影响**：存在文件但损坏/权限错误时，可能继续运行，且配置处于部分更新状态（依赖实现细节）。
  - **建议**：默认配置加载失败时应明确报错并退出，或回滚到加载前的配置状态。

### 建议的后续改进（按优先级）

- **高优先级**
  - 实现或删除 `enable_console_log`（让配置与行为一致）。
  - 提供 `sunkv.conf` 样例文件，并在 README 说明覆盖顺序（conf < CLI）。

- **中优先级**
  - 支持 `--log-file`（与 `log_file` 配置一致）。
  - `log_level` 解析大小写不敏感。
  - 将 `WAL/Snapshot` 的 `stderr DEBUG` 统一迁移到 `LOG_DEBUG`（或加总开关）。

- **低优先级**
  - 真正实现 section 语义的 INI（如确实需要）。
  - 让“性能压测模式”下自动压低热点日志（采样/降级）。
