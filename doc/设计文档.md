## 项目背景
当前主流 KV 存储系统如 Redis 提供了高性能、低延迟的数据访问能力，广泛应用于缓存、消息队列等场景。
本项目目标是：
从零实现一个高性能 KV 存储系统，用于深入理解：
- 高并发网络编程（Reactor）
- 存储引擎设计
- 持久化机制（WAL）
- 系统性能优化
## 项目目标
支持基础命令：
- GET
- SET
- DEL
支持 RESP 协议
支持 Pipeline（多命令连续处理）
支持多客户端并发访问
支持持久化（WAL + Snapshot）

性能目标：
- QPS ≥ 50,000（单机）
- 支持 ≥ 1,000,000 keys
- 平均延迟 < 1ms（本地测试）
- 支持 ≥ 1000 并发连接

稳定性目标：
- 崩溃后可恢复数据
- 无内存泄漏
- 长连接稳定运行

## 核心设计
### 网络层设计
模型设计：多线程 Reactor + one loop per thread
架构
```
Main Thread
   └── Acceptor（监听连接）

Worker Threads
   └── EventLoop（处理 IO）
```
关键点：
- 使用 epoll（ET 模式）
- 非阻塞 socket
- 每个线程独立 EventLoop
- 使用 eventfd 实现线程唤醒

为什么这样设计？
- 避免线程竞争
- 提高 CPU 利用率
- 减少锁开销
### 协议层设计
采用类似 Redis 的 RESP 协议：
示例：
```
*3
$3
SET
$3
key
$5
value
```

解析方式
使用状态机解析：
```
Array → BulkLen → Data → Done
```
设计优势
- 避免粘包问题
- 支持 pipeline
- 高性能（少拷贝）

### 存储引擎设计
数据结构

```
ShardedKVStore
 ├── Shard 1 (map + mutex)
 ├── Shard 2
 ├── ...
```
分片策略
```
shard_index = hash(key) % N
```
优点
- 降低锁竞争
- 提高并发性能
### 缓存淘汰（LRU）
结构：

```
unordered_map + 双向链表
```

操作复杂度：
- GET：O(1)
- SET：O(1)
- 淘汰：O(1)
### 持久化设计 

WAL（Write Ahead Log）
写入流程：
- SET → 写 WAL → 写内存

Snapshot
- 定期 dump 内存
- 启动时加载

恢复流程
- Load Snapshot
- Replay WAL

## 并发模型
策略
- IO 线程处理网络
- 存储使用分片锁

锁粒度
- 每个 Shard 一把 mutex
- 避免全局锁

关键点
- 不跨线程操作 fd
- 使用 runInLoop 保证线程安全
