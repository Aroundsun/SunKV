## 优化轮次结果（第 1 轮）
### 优化内容
- 目标：降低热点路径日志开销，不修改日志级别配置。
- 调整：
  - 连接建立/关闭日志由高频 `INFO` 下调为 `DEBUG`（仅热点路径）。
  - 移除消息处理路径中的十六进制 `stderr` 打印。
  - 连接等待关闭日志从每 100ms 一次降为每 1s 一次。

### 复测命令（与基线同口径）
- `redis-benchmark -h 127.0.0.1 -p 6393 -n 5000 -c 20 -P 1 -t set,get -q`
- `redis-benchmark -h 127.0.0.1 -p 6393 -n 5000 -c 20 -P 1 -t set,get --csv`

### 复测结果
- QPS:
  - SET: `22123.89 requests per second`
  - GET: `27777.78 requests per second`
- 延迟:
  - SET: avg `0.723 ms`, p50 `0.615 ms`, p95 `1.463 ms`
  - GET: avg `0.444 ms`, p50 `0.407 ms`, p95 `0.759 ms`

### 对比结论
- 相比基线，吞吐显著提升（约 7x~8x），延迟显著下降。
- 当前仍未达到 QPS 目标 `>= 50,000`，需继续优化。

## 优化轮次结果（第 2 轮）
### 优化内容
- 目标：降低命令处理路径的分配开销。
- 调整：
  - 将 `onMessage` 中 `RESPParser` 从每请求 `std::make_unique` 改为栈对象。

### 复测结果
- QPS:
  - SET: `23255.81 requests per second`
  - GET: `27624.31 requests per second`
- 延迟:
  - SET: avg `0.679 ms`, p50 `0.591 ms`, p95 `1.391 ms`
  - GET: avg `0.434 ms`, p50 `0.399 ms`, p95 `0.719 ms`

### 对比结论
- 相比第 1 轮有小幅提升（主要体现在 SET）。
- 当前瓶颈已不再是单一日志开销，后续需进入更深层（命令执行路径与数据访问并发）的优化。

## 优化轮次结果（第 3 轮）
### 优化内容
- 目标：提高并发处理能力（运行参数调优）。
- 调整：
  - 新增命令行参数支持：`--thread-pool-size`、`--max-connections`
  - 压测实例使用：
    - `--thread-pool-size 12`
    - `--max-connections 4000`

### 复测结果
- QPS:
  - SET: `28901.73 requests per second`
  - GET: `29940.12 requests per second`
- 延迟:
  - SET: avg `0.660 ms`, p50 `0.599 ms`, p95 `1.231 ms`
  - GET: avg `0.446 ms`, p50 `0.383 ms`, p95 `0.679 ms`
- 并发连接测试（c=1000）:
  - GET: `29239.77 requests per second`, p50 `16.735 ms`

### 对比结论
- 吞吐继续提升，已接近 `30k rps`，但仍未达到 `>= 50,000` 的阶段目标。
- 1000 并发连接能力稳定，但高并发下尾延迟仍偏高，后续应继续优化命令执行路径与连接管理路径。

## 优化轮次结果（第 4 轮）
### 优化内容
- 目标：减少命令分发长链上的无效比较成本。
- 调整：
  - 在 `Server::processCommand` 中为高频命令增加快速路径：
    - `SET / GET / DEL / EXISTS`
  - 让高频命令在进入长分支链前提前返回。

### 复测结果
- QPS（`-q` 输出）:
  - SET: `24509.80 requests per second`
  - GET: `27777.78 requests per second`
- 延迟与吞吐（`--csv` 输出）:
  - SET: rps `29940.12`, avg `0.579 ms`, p50 `0.519 ms`, p95 `1.103 ms`
  - GET: rps `32258.06`, avg `0.374 ms`, p50 `0.335 ms`, p95 `0.631 ms`
- 并发连接测试（c=1000）:
  - GET: `28409.09 requests per second`, p50 `17.439 ms`

### 对比结论
- 相比第 3 轮，常规并发下（c=20）SET/GET 吞吐继续提升，延迟进一步下降。
- 高并发（c=1000）吞吐接近第 3 轮，说明下一步瓶颈更偏向连接/并发路径而非纯命令分支比较。

## 优化轮次结果（第 5 轮）
### 优化内容
- 目标：优化连接生命周期与连接容器路径开销。
- 调整：
  - `TcpServer` 连接表由 `std::map` 改为 `std::unordered_map`。
  - 修正连接回调统计逻辑：在关闭阶段按连接状态对称更新 `current_connections_`，避免统计累积失真。

### 复测结果
- QPS（`-q` 输出）:
  - SET: `26737.97 requests per second`
  - GET: `31250.00 requests per second`
- 延迟与吞吐（`--csv` 输出）:
  - SET: rps `27027.03`, avg `0.673 ms`, p50 `0.623 ms`, p95 `1.247 ms`
  - GET: rps `32894.74`, avg `0.370 ms`, p50 `0.319 ms`, p95 `0.679 ms`
- 并发连接测试（c=1000）:
  - GET: `28248.59 requests per second`, p50 `17.231 ms`

### 对比结论
- 普通并发下 GET 吞吐再提升，延迟略有改善。
- 1000 并发下吞吐仍在 `~28k rps` 区间，瓶颈已转向更深层的数据访问与请求处理路径。

## 优化轮次结果（第 6 轮）
### 优化内容
- 目标：降低 GET 读路径中的额外写成本。
- 调整：
  - 在 GET 快速路径中，命中过期键时不在读请求内执行 `erase`，只返回空值。
  - 保持过期键清理由后台清理路径处理。

### 复测结果
- QPS（`-q` 输出）:
  - SET: `25252.53 requests per second`
  - GET: `31055.90 requests per second`
- 延迟与吞吐（`--csv` 输出）:
  - SET: rps `27173.91`, avg `0.618 ms`, p50 `0.559 ms`, p95 `1.191 ms`
  - GET: rps `28089.89`, avg `0.409 ms`, p50 `0.375 ms`, p95 `0.663 ms`
- 并发连接测试（c=1000）:
  - GET: `28901.73 requests per second`, p50 `17.215 ms`

### 对比结论
- 与第 5 轮相比，`c=1000` 下有轻微回升，改善有限。
- 当前主瓶颈更可能在高并发请求处理总体路径（而非单点锁写操作），后续需进一步做系统级 profiling（CPU 热点/锁竞争）。

## 优化轮次结果（读写锁完整版 A/B）
### 优化内容
- 目标：在保持可编译和一致语义的前提下，完成完整读写锁改造并做 A/B 验证。
- 调整：
  - `multi_storage_mutex_`、`simple_storage_mutex_` 统一改为 `std::shared_mutex`
  - 写路径统一使用 `std::unique_lock<std::shared_mutex>`
  - 读路径（例如 `GET/EXISTS/KEYS/DBSIZE` 等）使用 `std::shared_lock<std::shared_mutex>`

### 复测结果
- QPS（`-q` 输出）:
  - SET: `22727.27 requests per second`
  - GET: `30674.85 requests per second`
- 延迟与吞吐（`--csv` 输出）:
  - SET: rps `25510.21`, avg `0.707 ms`, p50 `0.639 ms`, p95 `1.335 ms`
  - GET: rps `29761.90`, avg `0.419 ms`, p50 `0.383 ms`, p95 `0.743 ms`
- 并发连接测试（c=1000）:
  - GET: `28248.59 requests per second`, p50 `17.983 ms`

### 对比结论
- 本轮“完整读写锁版”相较前一轮没有带来明显提升，部分指标略有回落。
- 说明当前瓶颈并非单纯由互斥锁类型决定；下一步应转向 CPU 热点与请求路径 profiling，再做定点优化。

## 锁策略回归验证（混合策略）

## Profiling 定位结果（受限环境下）
### 系统级采样尝试
- `perf` 与 `strace -p` 在当前环境均被内核策略限制（`perf_event_paranoid=4`，`ptrace` 禁止），无法直接做系统级火焰图采样。

### 进程内轻量埋点（已实施）
- 在 `onMessage` 路径增加两段计时：
  - RESP 解析耗时（`profile.avg_parse_us`）
  - 命令执行耗时（`profile.avg_process_us`）
- 采样负载：
  - `redis-benchmark -n 5000 -c 20 -P 1 -t get -q`
- 结果：
  - GET 吞吐：`34246.57 requests per second`（该轮）
  - `profile.avg_parse_us = 1.24458 us`
  - `profile.avg_process_us = 15.891 us`

### 结论
- 从当前埋点看，**命令执行阶段耗时显著高于解析阶段**（约 12.8 倍）。
- 下一轮优化优先级：
  1. `processCommand` 内高频命令链路（尤其 GET 路径）；
  2. 数据访问与响应构建中的对象/字符串处理成本；
  3. 再做同口径埋点复测确认收益。

## 优化轮次结果（第 7 轮：命令链路精简）

- 并发连接测试（c=1000）:
  - GET: `29585.80 requests per second`, p50 `16.463 ms`
- 埋点（DEBUG INFO）:
  - `profile.avg_parse_us = 1.39972`

### 对比结论

- 但 `profile.avg_process_us` 仍显著高于解析耗时，执行阶段依旧是主瓶颈，后续应继续针对响应构建与数据访问细化优化。
