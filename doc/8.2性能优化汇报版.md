# 8.2 性能优化汇报版（1页）

## 目标与现状

- 目标：
  - QPS >= 50,000
  - 延迟 < 1ms
  - 并发连接 >= 1000
- 当前（近期稳定区间）：
  - SET: 约 `24k~25k req/s`，p50 约 `1.6~1.8 ms`
  - GET: 约 `30k~31k req/s`，p50 约 `0.88~0.93 ms`
  - 结论：GET 接近目标线，SET 和总体吞吐仍未达标。

---

## 关键优化动作（已完成）

| 类别 | 已落地优化 | 效果结论 |
| --- | --- | --- |
| 热点降噪 | 高频日志降噪、移除调试输出 | 首轮收益最大，吞吐从 `3k` 级提升到 `2w+` |
| 命令路径 | 高频命令快速路径、重复分支删除 | 常规并发与高并发 GET 均有改善 |
| 发送路径 | `sendInLoop` 引入 `writev` 合并写 | 场景型收益，整体同量级波动 |
| 解析路径 | `RESPParser` 栈化 + `string_view` 输入 | 小幅稳定增益 |
| 并发路径 | `multi_storage` 读写锁细化（GET 共享锁） | GET/SET 均小幅正向 |
| 语义收敛 | 移除 `simple_storage` 旧路径，统一 `multi_storage` | 一致性与可维护性提升 |

---

## 关键量化样本（同口径）

压测口径：`redis-benchmark -n 50000 -c 50 -t set,get -q`

| 轮次样本 | SET | GET |
| --- | --- | --- |
| 批量解析轮次 | `23832` | `30750` |
| 读写锁细化轮次 | `24655` | `30902` |
| `RESPParser string_view` 轮次 | `25075` | `31095` |

趋势：进入“细优化阶段”，呈持续小幅改善，但离 `50k` 仍有差距。

---

## 经验结论

- **有效**：热点日志降噪、命令链路简化、分配/拷贝削减、批量解析。
- **不稳定/收益低**：一刀切全量读写锁、每次回调启停 TCP_CORK。
- 当前瓶颈已收敛到：`onMessage -> processCommand -> sendInLoop -> write` 的细粒度成本叠加。

---

## 下一步建议（8.3 前置参考）

1. 固定口径做 `>=3` 次压测并取中位数，形成稳定基线。  
2. 继续压缩高频路径中的临时对象与小包写成本。  
3. 在 `1000` 并发场景重点关注 p95/p99 与抖动，补齐稳定性指标。  

---

## 一句话结论

8.2 已完成从“粗优化”到“细优化”的过渡：系统性能显著优于基线，GET 基本逼近目标，但 SET 与总吞吐距离 `50k` 仍需后续阶段持续优化。
