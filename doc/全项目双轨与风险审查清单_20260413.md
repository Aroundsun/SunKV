# 全项目双轨与风险审查清单（2026-04-13）

本文档记录本轮对 SunKV 全项目的审查结论，重点覆盖双轨问题与高风险缺陷，按严重级别排序，便于后续逐项收敛。

## Critical

### 1) 写命令持久化路径不完整（存在数据丢失风险）
- **位置**：`server/Server.cpp`（`processCommand()`）
- **现象**：
  - `SET/DEL` 会写 WAL；
  - 但 `LPUSH/RPUSH/LPOP/RPOP/SADD/SREM/HSET/HDEL/EXPIRE/PERSIST/FLUSHALL` 等修改 `multi_storage_` 的路径大多未写 WAL。
- **风险**：
  - 异常退出或重启后，上述命令修改无法回放，导致数据不一致或丢失。
- **建议**：
  - 所有“会修改数据状态”的命令统一走单一路径（同一语义的 WAL 记录与内存修改流程）；
  - 为每类写命令补齐“重启后恢复一致性”回归测试。

### 2) TCP 服务启动路径重复且顺序不一致
- **位置**：`server/Server.cpp`（`initializeNetwork()`、`setupConnectionCallbacks()`）
- **现象**：
  - `initializeNetwork()` 已调用 `tcp_server_->start()`；
  - `setupConnectionCallbacks()` 里再次 `start()`；
  - 回调绑定发生在后者。
- **风险**：
  - 启动窗口内连接行为不确定；
  - “双入口启动”增加维护复杂度与潜在竞态。
- **建议**：
  - 收敛为单一路径：先设置回调，再且只调用一次 `start()`。

## High

### 3) 线程池存在双轨（Server 外部一套 + TcpServer 内部一套）
- **位置**：
  - `server/Server.cpp`（`thread_pool_`）
  - `network/TcpServer.cpp`（`threadPool_`）
- **现象**：
  - Server 自建并启动了一个线程池；
  - TcpServer 内部也有线程池并在 `start()` 中启动。
- **风险**：
  - `thread_pool_size` 配置语义可能不一致；
  - 资源浪费、关闭流程复杂、可观察性变差。
- **建议**：
  - 统一线程池来源，避免重复创建与双轨控制。

### 4) `SET` 接口仍保留旧形态参数语义，易造成理解偏差
- **位置**：`storage/WAL.h`、`storage/WAL.cpp`
- **现象**：
  - 底层已收敛到 `DataValueCodec`；
  - 但仍有 `write_set(key, string, ttl_ms)` 旧风格接口。
- **风险**：
  - 接口表意与实现语义可能被误解，后续易再次引入分叉实现。
- **建议**：
  - 明确主语义为 `write_set(key, DataValue)`；
  - 将 string 版本标注为便捷封装（或后续收敛移除）。

## Medium

### 5) 独立恢复器与主恢复路径仍有语义分叉迹象
- **位置**：`storage/Recovery.cpp` vs `server/Server.cpp`
- **现象**：
  - 主恢复路径以多类型为中心；
  - `DataRecovery` 仍以 string map 视角为主（虽已接入 codec 解码分支）。
- **风险**：
  - 后续维护可能出现“主路径已改，恢复器未同步”的双轨回归。
- **建议**：
  - 明确 `DataRecovery` 的角色（离线工具或主路径）；
  - 若继续保留，建议与主路径语义完全对齐并加一致性测试。

## 需确认项

### 6) 命令落盘覆盖率缺少自动化守护
- **位置**：`server/Server.cpp`（大量命令分支）
- **现象**：
  - 写命令分支较多，人工维护容易遗漏 WAL 记录。
- **建议**：
  - 增加“命令 -> 是否落 WAL”的表驱动审计测试；
  - 在 CI 中对关键写命令执行“重启恢复一致性”校验。

### 7) 时间戳语义与文档需持续对齐
- **位置**：`storage/WAL.*`、`storage/Snapshot.*`、相关文档
- **现象**：
  - 持久化编码与恢复过滤依赖时间戳语义，历史演进频繁。
- **建议**：
  - 将当前时间戳语义写入统一设计文档；
  - 通过测试锁定“快照 + WAL 过滤”的行为边界。

---

## 本轮优先处理建议
1. 先修复 **写命令持久化不完整**（Critical-1）  
2. 再收敛 **启动路径双轨**（Critical-2）  
3. 最后统一线程池控制与恢复器语义（High/Medium）
