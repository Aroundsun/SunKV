# SunKV 存储模块 v2（从零设计）v0.1

> 目标：推倒重来存储模块，做到高扩展、低耦合、可插拔后端、持久化事件化、缓存装饰器化。
> 本文不考虑与现有实现/数据格式兼容性（你要求的）。

---

## 1. 背景与问题陈述

当前 storage 目录存在的问题（抽象层面）：

- 存储职责混杂：命令语义、数据结构、TTL、持久化触发、快照/WAL编排交叉在一起。
- 功能模块“名义存在但未接入”：缓存、分片等模块未成为主链路的一部分。
- 抽象边界过宽：接口既承载存储抽象，又承载命令语义、甚至为 WAL 预留输出参数，导致牵一发动全身。
- 不利于演进：未来要接入真实分片、缓存、磁盘后端、事务/批量/一致性增强时会持续膨胀。

因此需要一个“从零”的存储架构：分层清晰、单向依赖、能力以装饰器/后端替换方式接入。

---

## 2. 设计目标（Must）

- **低耦合**：Server/协议层不直接依赖 WAL/Snapshot/具体存储实现细节。
- **高扩展性**：
  - 可替换后端：单机内存 / 真分片 /（未来）磁盘后端
  - 可插拔能力：缓存、统计、限流、审计
- **一致性与可测试**：
  - 命令语义在 API 层可单测，不启动网络
  - 持久化由事件驱动，不散落在命令分支
- **可观测**：统一返回状态码；统计/埋点通过装饰器实现。

---

## 3. 非目标（Not now）

- 分布式一致性（Raft、多副本）
- 跨 key 强事务（multi-key atomic）
- 复杂内存淘汰策略（先留扩展点）
- 兼容旧快照/WAL 版本

---

## 4. 总体架构（分层）

单向依赖（禁止反向）：

Server/Protocol
  -> storage2/api (命令语义入口)
    -> storage2/engine (执行引擎/并发/过期处理)
      -> storage2/backend (可替换后端)
    -> storage2/persistence (事件消费：WAL/Snapshot)
  -> storage2/decorators (可选装饰器：cache/metrics/audit)

关键原则：
- **存储层不认识 RESP**：不返回协议串。
- **持久化不认识命令**：只看 Mutation/Record 编解码。
- **缓存不侵入核心**：用装饰器包装 API。

## 6. 数据模型（核心）

### 6.1 Record（唯一真相）

- key: string
- value: DataValue（STRING/LIST/SET/HASH）
- expire_at_us: int64（epoch microseconds，-1 = no ttl）
- version: uint64（可选，便于未来实现乐观并发/快照一致性）

建议：

struct Record {
  DataValue value;
  int64_t expire_at_us;   // -1 no ttl
  uint64_t version;       // monotonic
};

TTL 语义统一：只以 expire_at_us 判断过期。

### 6.2 Mutation（写入事件）

写操作必须产出 Mutation，由持久化层统一消费。

enum MutationType {
  PutRecord,   // key -> record（包含 value + expire_at）
  DelKey,      // 删除 key
  ClearAll,    // 清空
};

struct Mutation {
  MutationType type;
  std::string key;                 // ClearAll 可为空
  std::optional<Record> record;    // PutRecord 必填
  int64_t ts_us;                   // 事件时间
};

注意：命令层不调用 WAL；只返回 Mutation。

## 7. API 设计（命令语义层）

### 7.1 统一返回

- StatusCode：Ok / NotFound / WrongType / InvalidArg / InternalError
- StorageResult<T>：包含 status + value + mutations（MutationBatch）

示例：

StorageResult<std::optional<std::string>> Get(key)
StorageResult<int64_t> LPush(key, values)
StorageResult<int64_t> HSet(key, field, value)

### 7.2 命令集合（第一阶段最小闭环）

- string：SET/GET/DEL/EXISTS
- list：LPUSH/RPUSH/LPOP/RPOP/LLEN
- set：SADD/SREM/SCARD/SISMEMBER/SMEMBERS
- hash：HSET/HGET/HDEL/HLEN/HEXISTS/HGETALL
- ttl：EXPIRE/PERSIST/TTL/PTTL
- misc：DBSIZE/KEYS（可后做 scan）

说明：
- WrongType 由 API 层统一判断返回，Server 只把 StatusCode 映射到 RESP。
- 不要求跨 key 原子；单 key 原子必须保证。

## 8. Engine 设计（执行、并发、过期）

### 8.1 并发模型（第一版）

Backend 采用分片锁（shared_mutex）：

- 单 key 操作锁 1 个 shard
- multi-key（未来）按 shard id 排序加锁防死锁

### 8.2 过期策略（统一）

- 惰性删除：任何访问（读/写）先检查过期；过期则删除并视为 NotFound。
- 后台清理：可选线程，按 shard 扫描；只影响空间占用，不改变语义。

### 8.3 类型转换策略（明确规则）

- 默认不允许字符串自动变容器（更严格）或允许（Redis-like）。
- 如果要 Redis-like：当 key 不存在时创建对应容器；当 key 为 STRING 且执行 LPUSH/SADD/HSET 时，转为目标类型并 **不注入旧 string_value**。
- 这一条必须写成契约并单测。

## 9. Backend 设计（可替换）

### 9.1 IBackend（最小接口）

- GetRecord(key) -> optional<Record>
- PutRecord(key, Record)
- DelKey(key)
- ClearAll()
- IterateKeys(callback)（用于快照/keys/dbsize）

Backend 不暴露命令语义，只暴露 Record CRUD。

### 9.2 InMemoryBackend

- unordered_map<string, Record> + lock
- 简单、稳定、作为基线实现

### 9.3 ShardedBackend（真分片）

- N shards，每 shard 独立 map + lock
- key 哈希取 shard
- 真实分片必须满足：不同 shard 之间无共享存储实例（避免“伪分片”）

---

## 10. Persistence 设计（完全解耦）

### 10.1 PersistenceOrchestrator

职责：
- 接收 MutationBatch（同步或异步队列）
- 写 WAL（顺序、刷盘策略）
- 周期/触发快照
- 启动恢复：Snapshot -> WAL replay -> Engine.loadSnapshot(records)

关键点：
- Server 不感知 WAL/Snapshot 存在与否
- Engine 不直接写文件

### 10.2 编解码

- RecordCodec：对 Record 的 value 使用 DataValueCodec（你现有可复用/升级）
- WALCodec（WalCodec）：对 Mutation 序列化（type + key + record bytes + ts），并保证读写同源

---

## 11. Cache 设计（装饰器）

- CachedStorageAPI wraps IStorageAPI
- read-through + write-through（第一版）
- v0.1 实现约束（安全优先）：
  - 仅缓存 `GET(key)` 的“无 TTL”结果（`PTTL(key) == -1`）
  - 所有可能影响 key 的写操作都会 invalidate(key)
- 缓存策略（LRU/LFU/ARC）统一为一套枚举与工厂
- 缓存不应改变语义：TTL/类型错误必须与底层一致

---

## 12. Wiring（组装方式）

主程序启动时构建（推荐顺序：越外层越“横切能力”）：

backend = ShardedBackend(N) or InMemoryBackend
engine  = StorageEngine(backend)
api     = engine（或 api wraps engine）
api     = CacheDecorator(api)     // 可选
api     = MetricsDecorator(api)   // 可选
persist = PersistenceOrchestrator(options) // 可选（WAL/Snapshot）
api     = OrchestratedStorageAPI(api, persist) // 可选：自动提交 mutations

Server 只持有 IStorageAPI*。

### 12.1 Factory（推荐）

v0.1 已实现 `storage2/Factory`：一次性组装 backend/engine/decorators/persistence。

约定（重要）：
- 当启用 `OrchestratedStorageAPI` 时：上层拿到的 `StorageResult<T>` 中 `mutations` 会被清空（避免重复提交）。
- `PersistenceOrchestrator::flush()` 语义：尽力 drain 当前已提交队列并落盘（用于测试/退出路径）。

## 13. 验收标准（定义“解耦成功”）

- Server 源码中不出现 WAL/Snapshot include 与调用（0 次）
- 更换 backend（single <-> sharded）不改 Server，不改持久化代码
- 关闭 WAL/快照不改 Server，不改命令语义实现
- 缓存开关只改 wiring，不改业务逻辑
- 单测可直接针对 StorageEngine/API 运行，覆盖命令语义与 TTL
