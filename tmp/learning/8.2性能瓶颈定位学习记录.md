# 8.2 性能瓶颈定位学习记录

## 目标
在不盲目改代码的前提下，先回答三个问题：
1. 时间主要花在 CPU、I/O、还是锁等待？
2. 慢在网络层、协议层、命令分发层、还是存储层？
3. 哪一类请求最影响吞吐和尾延迟（p95/p99）？

## 方法一：先做“最小变量”基线
1. 固定压测命令（例如只测 GET）。
2. 固定并发（例如 c=100），只改一个变量（如 pipeline 从 1 到 8）。
3. 每次只观察 3 个指标：`rps`、`p50`、`p95`。

判定思路：
- `rps` 上不去且 `p95` 明显变大：通常有排队或锁竞争。
- `rps` 变化不大但 `p50/p95` 都高：常见于每请求固定开销过大（日志、序列化、系统调用）。

## 方法二：分层排除法
按从外到内的顺序逐层排查：
1. 网络层：连接数、收发吞吐、是否有大量短连接抖动。
2. 协议层：解析/序列化是否成为热点。
3. 命令层：参数校验、命令分发、对象创建是否过重。
4. 存储层：锁粒度、数据结构命中、持久化写入开销。

实践建议：
- 先跑 `PING` 和 `GET` 对比：`PING` 慢说明网络/框架开销大；`GET` 远慢于 `PING` 说明存储链路更重。

## 方法三：日志与统计交叉验证
1. 保留服务端运行日志（本项目按运行分文件）。
2. 压测结束后检查：
   - 是否存在“等待连接关闭”时间过长。
   - 是否有异常重连、错误响应、线程退出异常。
3. 对比压测结果和服务端日志时间段，确认慢点是否一致。

关键点：
- 只看压测数字不够，必须和服务端日志时间线对应。

## 方法四：热点采样（进阶）
在 Linux 上可用 `perf`（或等价工具）做采样，关注热点函数占比：
- 如果热点集中在日志格式化/输出：优先优化日志路径。
- 如果热点集中在锁与容器操作：优先优化并发结构与访问模式。
- 如果热点集中在内存分配：优先减少临时对象和分配次数。

## 方法五：形成“改动-验证”闭环
每一轮优化都记录四件事：
1. 改了什么（只改一个主要点）。
2. 为什么改（预期改善哪个指标）。
3. 复测结果（同一组压测命令）。
4. 结论（有效/无效/副作用）。

这样可以避免“改了很多，但不知道哪个有效”。

## 实战记录：优化第 1 轮（日志热点降噪）

### 现象
- 基线仅约 `3k rps`，但服务器日志文件快速膨胀，且大量行来自连接事件与消息路径打印。

### 假设
- 高并发下，非必要高频日志（尤其 `stderr` 打印）成为主要固定开销，拖慢每请求处理。

### 改动
1. `server/Server.cpp`
   - `onConnection/onDisconnection` 从 `LOG_INFO` 改为 `LOG_DEBUG`
   - 删除 `onMessage` 中每请求十六进制 `stderr` 打印
   - “等待连接关闭”日志从 100ms 降为 1s 采样
2. `network/TcpServer.cpp`
   - 新建连接/移除连接日志从 `LOG_INFO` 改为 `LOG_DEBUG`

### 结果
- QPS：
  - SET：`3180.66 -> 22123.89`
  - GET：`3198.98 -> 27777.78`
- 延迟：
  - SET avg：`6.271ms -> 0.723ms`
  - GET avg：`6.183ms -> 0.444ms`

### 结论
- 假设成立：热点路径日志确实是显著瓶颈之一。
- 下一轮应继续看：
  - 命令执行路径对象分配（`RESPParser` 每请求 new）
  - 连接关闭与生命周期管理开销
  - 存储路径锁竞争与数据结构命中

## 实战记录：优化第 2 轮（去除 parser 堆分配）

### 改动
- `server/Server.cpp` 中 `onMessage`：
  - `auto parser = std::make_unique<RESPParser>();` -> `RESPParser parser;`

### 结果
- QPS：
  - SET：`22123.89 -> 23255.81`
  - GET：`27777.78 -> 27624.31`（基本持平，测量波动范围内）
- 延迟：
  - SET avg：`0.723ms -> 0.679ms`
  - GET avg：`0.444ms -> 0.434ms`

### 结论
- 堆分配消除带来小幅收益，但不是决定性瓶颈。
- 下一轮更可能在：
  - 命令匹配分支与字符串处理开销
  - 共享数据结构访问与锁竞争

## 实战记录：优化第 3 轮（运行参数并发调优）

### 改动
1. `common/Config.cpp`
   - 增加命令行参数解析：
     - `--thread-pool-size`
     - `--max-connections`
   - 更新 `printUsage()` 帮助信息
2. 压测启动参数调整：
   - `--thread-pool-size 12`
   - `--max-connections 4000`

### 结果
- QPS：
  - SET：`23255.81 -> 28901.73`
  - GET：`27624.31 -> 29940.12`
- 延迟：
  - SET avg：`0.679ms -> 0.660ms`
  - GET avg：`0.434ms -> 0.446ms`（小幅波动）
- 1000 并发连接：
  - GET：`29239.77 rps`，p50 `16.735ms`

### 结论
- 并发能力调优有效，吞吐进一步抬升到约 `30k rps`。
- 但距离 `50k` 目标仍有差距，下一轮应优先分析：
  - 命令处理链路中的分支与字符串处理开销
  - 高并发关闭阶段与连接容器操作的成本

## 实战记录：优化第 4 轮（高频命令快速路径）

### 改动
- 在 `Server::processCommand` 里将高频命令提前：
  - `SET / GET / DEL / EXISTS`
- 这些命令在进入长分支链前优先匹配并返回。

### 结果
- `-q` 输出：
  - SET：`24509.80 rps`
  - GET：`27777.78 rps`
- `--csv` 输出（更稳定）：
  - SET：`29940.12 rps`，avg `0.579ms`
  - GET：`32258.06 rps`，avg `0.374ms`
- `c=1000`：
  - GET：`28409.09 rps`，p50 `17.439ms`

### 结论
- 低并发到中并发下效果明显，说明分支链开销确有影响。
- 高并发提升有限，下一轮应聚焦连接管理与共享结构并发成本。

## 实战记录：优化第 5 轮（连接路径与容器优化）

### 改动
1. `network/TcpServer.h`
   - `connections_` 从 `std::map` 改为 `std::unordered_map`
2. `server/Server.cpp`
   - `onConnection` 按 `conn->connected()` 区分连接建立/关闭，做对称计数
   - 规避关闭阶段 `current_connections_` 统计失真

### 结果
- `-q` 输出：
  - SET：`26737.97 rps`
  - GET：`31250.00 rps`
- `--csv` 输出：
  - SET：`27027.03 rps`，avg `0.673ms`
  - GET：`32894.74 rps`，avg `0.370ms`
- `c=1000`：
  - GET：`28248.59 rps`，p50 `17.231ms`

### 结论
- 容器与连接统计路径优化对常规并发有正收益。
- 高并发收益有限，后续需更聚焦：
  - 数据访问锁竞争
  - 请求处理路径中的对象与字符串操作成本

## 实战记录：优化第 6 轮（GET 读路径去写化）

### 改动
- `server/Server.cpp` 的 GET 快速路径中：
  - 命中过期键时不再在读请求里执行 `erase`
  - 只返回空值，过期清理由后台路径完成

### 结果
- `-q`：
  - SET：`25252.53 rps`
  - GET：`31055.90 rps`
- `--csv`：
  - SET：`27173.91 rps`，avg `0.618ms`
  - GET：`28089.89 rps`，avg `0.409ms`
- `c=1000`：
  - GET：`28901.73 rps`，p50 `17.215ms`

### 结论
- 高并发指标仅小幅波动，说明这个点不是决定性瓶颈。
- 下一步要获得明显提升，需要进入 profiling：
  - CPU 热点函数占比
  - 锁等待与争用热点

## 实战记录：读写锁完整版（A/B 验证）

### 背景
- 之前半途回退读写锁是因为改动面大导致编译错误。
- 本轮按“完整替换”执行并强制编译通过，再做同口径压测。

### 改动
1. `server/Server.h`
   - `multi_storage_mutex_`、`simple_storage_mutex_` 切换到 `std::shared_mutex`
2. `server/Server.cpp`
   - 所有对应写路径锁改为 `std::unique_lock<std::shared_mutex>`
   - 读路径改为 `std::shared_lock<std::shared_mutex>`

### 结果
- `-q`：
  - SET：`22727.27 rps`
  - GET：`30674.85 rps`
- `--csv`：
  - SET：`25510.21 rps`，avg `0.707ms`
  - GET：`29761.90 rps`，avg `0.419ms`
- `c=1000`：
  - GET：`28248.59 rps`，p50 `17.983ms`

### 结论
- 读写锁改造并未带来预期收益，甚至有轻微回落。
- 推断：当前主要瓶颈不在锁类型本身，而在请求处理与 CPU 热点路径。

## 实战记录：锁策略回归（混合策略）

### 背景
- 你提出“是否应该全换读写锁”，结论是不应一刀切。
- 因此做了回归验证：把锁恢复为 `mutex`，保留其他有效优化，观察整体表现。

### 结果
- `-q`：
  - SET：`25906.73 rps`
  - GET：`30120.48 rps`
- `--csv`：
  - SET：`22935.78 rps`，avg `0.783ms`
  - GET：`28901.73 rps`，avg `0.397ms`
- `c=1000`：
  - GET：`27777.78 rps`，p50 `17.471ms`

### 结论
- 混合策略在当前实现下更稳妥，开发复杂度更低，性能不劣于全量读写锁。
- 下一步优先级应回到 profiling，而不是继续在锁类型上做大范围替换。

## 实战记录：Profiling（受限环境 fallback）

### 1) 系统级工具受限
- `perf record` 失败：`perf_event_paranoid=4`
- `strace -p` 失败：`ptrace ... 不允许的操作`

这说明在当前机器权限下不能直接附加采样。

### 2) 采用进程内轻量埋点
- 在 `Server::onMessage` 加两个计时累加器：
  - `profile_parse_ns_`：`RESPParser::parse` 段
  - `profile_process_ns_`：`processCommand` 段
- 并在 `DEBUG INFO / STATS` 输出均值（微秒）。

### 3) 实测数据
- 压测：`redis-benchmark -n 5000 -c 20 -P 1 -t get -q`
- 吞吐：`34246.57 rps`
- 埋点输出：
  - `profile.avg_parse_us = 1.24458`
  - `profile.avg_process_us = 15.891`

### 4) 结论与学习点
- 解析开销不是主瓶颈，命令执行链路开销更大（约 `12.8x`）。
- 在受限权限环境下，**“应用内埋点 + 同口径压测”**是可行的 profiling 替代方案。

## 实战记录：优化第 7 轮（删除重复命令分支）

### 改动
- 在 `processCommand` 中发现 `SET/GET/DEL/EXISTS` 存在“快速路径 + 后置重复路径”双份实现。
- 删除后置重复块，仅保留前置快速路径。

### 结果
- `-q`：
  - SET：`25380.71 rps`
  - GET：`30120.48 rps`
- `--csv`：
  - SET：`25906.73 rps`，avg `0.703ms`
  - GET：`31847.13 rps`，avg `0.387ms`
- `c=1000`：
  - GET：`29585.80 rps`，p50 `16.463ms`
- 埋点：
  - `profile.avg_parse_us=1.39972`
  - `profile.avg_process_us=22.0036`

### 结论
- 删除重复分支对高并发 GET 有正收益。
- 命令执行耗时仍远高于解析，下一步应继续压缩 process 路径中的对象/字符串开销。
